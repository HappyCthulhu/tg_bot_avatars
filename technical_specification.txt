Тестовое задание: Чат-бот с ИИ-аватарами (Aiogram 3.x)
Легенда
Вам предстоит разработать ядро сервиса для общения с ИИ-персонажами. Пользователь выбирает себе аватара (например, "Учёный кот", "Саркастичный подросток" или "Мудрый старец") и начинает с ним диалог. Главная "фишка" бота — он должен помнить не только
 последние сообщения (краткосрочная память), но и детали, которые всплывали в старых диалогах (долгосрочная память), чтобы ссылаться на них в будущем.

Цель задания
Разработать Telegram-бота на базе библиотеки Aiogram 3.x, который взаимодействует с пользователем от лица выбранного аватара, используя любую LLM (можно использовать OpenAI GPT, GigaChat, YandexGPT или локальную модель через Ollama). Критически важна
 реализация двух типов памяти.

Стек (обязательный)
    • Python 3.10+
    • Aiogram 3.x
    • Библиотека для работы с LLM (openai, aiohttp или requests)
    • База данных (предпочтительно PostgreSQL или SQLite для простоты запуска)
Функциональные требования
1. Выбор аватара (Сценарий /start)
    • При запуске бота командой /start пользователю показывается приветствие и клавиатура (Inline Keyboard) с 2-3 вариантами аватаров.
    • Каждый аватар должен иметь:
        ◦ Название (например, "Профессор истории").
        ◦ Краткое описание характера (System Prompt), которое влияет на стиль ответов.
    • При нажатии на кнопку — начинается диалог с этим аватаром. Бот должен запомнить, какой аватар выбран для текущего пользователя.
2. Общение с ИИ
    • Пользователь пишет сообщение, бот отправляет его в LLM с учетом системного промпта (характер аватара) и контекста беседы.
    • Ответ от ИИ должен приходить частями (стримингом) — это важно для оценки UX. Сообщение должно редактироваться (или отправляться новыми частями), пока не сформируется полностью.
3. Система памяти (Ключевая часть задания)
У бота должно быть два уровня памяти:
А. Краткосрочная память (Short-term / Dialog Memory)
    • Хранит последние N (например, 5-10) сообщений диалога (роль пользователя и роль ассистента).
    • Используется для поддержания контекста текущего разговора.
    • Реализация: Добавлять сообщения в список/очередь и передавать их в промпт LLM как историю.
Б. Долгосрочная память (Long-term / Fact Memory)
    • Бот должен уметь выделять и запоминать ключевые факты из диалога навсегда (для данного пользователя и аватара).
    • Механизм работы: Через каждые 3-5 сообщений или в фоне нужно делать дополнительный вызов LLM (или простую эвристику) с промптом: "Выдели важные факты о пользователе или теме диалога, которые нужно запомнить надолго (имя, интересы, события). Вер
ни JSON-список строк".
    • Полученные факты сохраняются в базу данных (таблица memory_facts).
    • Инъекция в контекст: При каждом новом обращении к LLM, бот достает из БД все факты, связанные с этим пользователем и аватаром, и добавляет их в начало промпта в виде строки: "Важно: ты помнишь об этом пользователе следующее: {список фактов}".
4. Работа с историей
    • Реализовать команду /history, которая выводит пользователю последние 10 сообщений из текущего диалога.
    • Реализовать команду /facts, которая выводит все факты, которые бот запомнил о пользователе (из долгосрочной памяти).
5. Дополнительно (Плюс к карме)
    • Сброс контекста: Команда /reset очищает краткосрочную историю диалога, но сохраняет долгосрочные факты.
    • Смена аватара: Возможность сменить аватара без перезапуска бота (команда /change_avatar возвращает к выбору).
Технические требования к реализации
    1. FSA (Конечные автоматы): Использовать aiogram.fsm для управления состояниями диалога (выбор аватара -> режим общения).
    2. База данных: Любая ORM (SQLAlchemy, SQLModel) или чистый SQL. Должны быть минимум 3 таблицы:
        ◦ users (user_id, текущий аватар).
        ◦ avatars (id, name, system_prompt).
        ◦ memory_facts (id, user_id, avatar_id, fact_text, created_at).
    3. Обработка ошибок: Бот не должен падать при ошибках API LLM. Пользователю должно отправляться понятное сообщение ("Извините, ИИ перегружен, попробуйте позже").
    4. Конфигурация: Ключи API и настройки должны читаться из переменных окружения (файл .env).
Критерии оценки
    1. Архитектура (30%):
        ◦ Чистота кода, разделение на модули (handlers, services, database, keyboards).
        ◦ Правильное использование FSM.
        ◦ Типизация (type hints) — приветствуется.
    2. Функциональность (40%):
        ◦ Работает ли краткосрочная память (очередь сообщений).
        ◦ Работает ли долгосрочная память (сохранение и подгрузка фактов).
        ◦ Корректно ли работает стриминг ответов от ИИ.
    3. Креативность (15%):
        ◦ Насколько интересно прописаны промпты аватаров.
        ◦ Как кандидат решил проблему парсинга фактов из ответа LLM (надежность JSON).
    4. UX/UI (15%):
        ◦ Удобство навигации (кнопки вместо команд).
        ◦ Информативность сообщений.
Что нужно прислать на проверку?

    2. Файл README.md с инструкцией по запуску (установка зависимостей, настройка .env).
    3. Видео (или скриншоты), демонстрирующее работу бота, особенно момент, где он вспоминает факт из долгосрочной памяти после перезапуска диалога.
